## 2. 项目背景与系统定位

本项目基于宇树 GO2 EDU 四足机器人，目标是实现厂房/园区巡检机器人系统。当前阶段并不是直接完成完整导航、建图、避障和图像识别，而是先完成机器人端与后台系统之间的最小通信闭环。

GO2 背部安装了 NVIDIA Jetson Orin 拓展坞。Jetson 作为机器人端中转 PC / 边缘计算单元，负责连接 GO2 本体和后台系统。

整体通信架构为：

```text
GO2 本体
    ↓ 192.168.123.x 内部有线网络 / Unitree DDS / CycloneDDS
Jetson Orin 拓展坞
    ↓ WebSocket / HTTP
后台服务器 / 笔记本 FastAPI Demo
```

当前阶段遵循以下原则：

1. GO2 原生 Unitree DDS / CycloneDDS 话题只在 GO2 与 Jetson 的内部有线网络中流转；
2. 后台不直接订阅 ROS2 DDS；
3. Jetson 作为机器人端中转站；
4. 后台只接收低频状态、任务进度、识别结果等信息；
5. 后台只下发开始、暂停、继续、停止、急停等低频命令；
6. 后续安全停车、急停、障碍物判断等逻辑应优先在 Jetson 本地完成，而不是完全依赖后台实时控制。

## 4. 开发方式调整

原计划是在笔记本 Docker 中搭建 Ubuntu 20.04 + ROS2 Foxy 环境，用于编写和初步检查源码。但实际使用过程中，VS Code Dev Containers、Codex 插件、Docker 权限、网络和代理等问题较多。

当天最终调整为更稳定的方式：

```text
Ubuntu 22.04 笔记本本地写代码
    ↓ rsync / scp 同步源码
Jetson Ubuntu 20.04 + ROS2 Foxy 原生编译运行
```

这意味着后续主流程不再依赖 Docker 或 Codex 插件，而是：

1. 笔记本负责写代码、管理工程、运行 FastAPI 后台 Demo；
2. Jetson 负责原生 ROS2 Foxy 编译和运行机器人端节点；
3. 两端通过 `scp`、`rsync` 或 Git 进行代码同步；
4. 避免在多个容器、多个目录中维护不同版本代码，减少文件丢失和版本混乱。

## 6. 已实现的最小通信闭环

当前已实现第一版最小闭环：

```text
ROS2 topic /inspection_state
    ↓
Jetson backend_client_node
    ↓ WebSocket
笔记本 FastAPI server.py
    ↓ HTTP command 接口
Jetson backend_client_node
    ↓
ROS2 topic /backend_command
```

这一闭环证明：Jetson 可以把 ROS2 内部状态转发给后台，后台也可以把命令下发回 Jetson 并发布到 ROS2 内部话题。

## 7. ROS2 侧设计

`backend_client_node` 实现了两个核心 ROS2 话题。

### 7.1 订阅状态话题

订阅话题：

```text
/inspection_state
```

消息类型：

```text
std_msgs/String
```

作用：

```text
模拟机器人当前巡检状态，例如 IDLE、PATROLLING、PAUSED、ERROR 等。
```

### 7.2 发布后台命令话题

发布话题：

```text
/backend_command
```

消息类型：

```text
std_msgs/String
```

作用：

```text
将后台下发的命令转发到 ROS2 内部，供后续任务控制节点、安全控制节点或巡检状态机订阅。
```

### 7.3 节点运行情况

当天在 Jetson 上运行：

```bash
ros2 run go2_backend_bridge backend_client_node
```

节点启动日志中出现：

```text
[INFO] [backend_client_node]: backend_client_node initialized
```

说明 `backend_client_node` 已经能够被 ROS2 找到并正常启动。

## 8. WebSocket 侧设计

Jetson 节点连接笔记本后台：

```text
ws://192.168.123.99:8000/ws/robot/GO2_001
```

连接成功后，节点完成以下动作：

1. 发送 `register` 注册消息；
2. 每 1 秒上传一次 `status`；
3. 接收后台 `command`；
4. 将 command 发布到 `/backend_command`；
5. 回传 `command_ack` 给后台；
6. WebSocket 断开后自动重连。

## 11. 当天遇到并解决的问题

### 11.1 FastAPI 后端启动失败

在笔记本本地启动 FastAPI 后端时，曾执行：

```bash
python3 -m uvicorn server:app --host 0.0.0.0 --port 8000
```

出现错误：

```text
ERROR: Error loading ASGI app. Attribute "app" not found in module "server".
```

问题含义：

- `uvicorn server:app` 表示在 `server.py` 文件中寻找名为 `app` 的 FastAPI 对象；
- 报错说明当前 `server.py` 中没有定义 `app`，或者当前目录下的 `server.py` 不是预期文件；
- 也可能是笔记本本地代码版本较旧，与 Jetson/GO2 侧代码不同步。

处理结论：

1. 统一笔记本本地和 Jetson/GO2 端代码；
2. 后端代码中必须有类似结构：

```python
from fastapi import FastAPI

app = FastAPI()
```

---

### 11.2 `ros2 run` 找不到可执行文件

曾遇到：

```text
No executable found
```

排查发现：

1. `backend_client_node.py` 曾经没有 `main()` 函数；
2. `setup.cfg` 没配置正确时，可执行文件会装到 `bin/`，而不是 ROS2 期望的 `lib/go2_backend_bridge/`。

最终通过以下配置解决。

`setup.py` 中必须有：

```python
entry_points={
    'console_scripts': [
        'backend_client_node = go2_backend_bridge.backend_client_node:main',
    ],
},
```

`setup.cfg` 中必须有：

```ini
[develop]
script_dir=$base/lib/go2_backend_bridge

[install]
install_scripts=$base/lib/go2_backend_bridge
```

### 11.8 VS Code Remote-SSH 下载 VS Code Server 很慢

当天遇到：

```text
Downloading VS Code Server...
```

下载很慢。

结合 NoMachine 下载失败的现象，判断根因大概率是：

```text
GO2/Jetson 远程端无法正常访问外网或无法解析域名
```

之前下载 NoMachine 时出现：

```text
Temporary failure in name resolution
unable to resolve host address ‘www.nomachine.com’
```

因此 VS Code Remote-SSH 让远程端自动下载 VS Code Server 时，也会卡住或失败。

推荐解决方式是在 VS Code 本地设置中加入：

```json
{
  "remote.SSH.localServerDownload": "always"
}
```

这样 VS Code 会优先在本地电脑下载 VS Code Server，再通过 SSH 上传到远程端，而不是让 GO2/Jetson 自己联网下载。

如果之前安装失败，可在远程端清理：

```bash
rm -rf ~/.vscode-server
rm -rf ~/.vscode-remote
```

然后重新连接。

### 12.4 NoMachine

NoMachine 适合完整远程桌面。

整体思路：

```text
GO2/Jetson：安装 NoMachine 服务端
笔记本：安装 NoMachine 客户端
```

连接参数通常为：

```text
Host: 192.168.123.18
Protocol: NX
Port: 4000
Username: unitree
```

由于 Jetson 不能直接下载，推荐流程是：

1. 在笔记本下载 ARM64 / aarch64 的 NoMachine `.deb` 包；
2. 通过 `scp` 传到 Jetson：

```bash
scp nomachine_arm64.deb unitree@192.168.123.18:~/
```

3. SSH 到 Jetson：

```bash
ssh unitree@192.168.123.18
```

4. 安装：

```bash
sudo dpkg -i ~/nomachine_arm64.deb
```

5. 如果缺依赖：

```bash
sudo apt -f install -y
```

6. 检查服务：

```bash
sudo systemctl status nxserver
```

7. 如未启动：

```bash
sudo systemctl start nxserver
sudo systemctl enable nxserver
```

8. 检查端口：

```bash
ss -lntp | grep 4000
```

原因：

## 14. 当天主要问题与解决思路汇总

| 问题 | 现象 | 判断 | 处理思路 |
|---|---|---|---|
| FastAPI 启动失败 | `Attribute "app" not found` | `server.py` 中缺少 `app` 或代码版本不一致 | 同步两端代码，确保定义 `app = FastAPI()` |
| `ros2 run` 找不到节点 | `No executable found` | Python 包入口或安装路径配置不正确 | 补充 `main()`、配置 `setup.py` 与 `setup.cfg` |
| Jetson 缺少依赖 | `websockets` distribution not found | Jetson 未安装 websockets | 下载 `cp38 + aarch64` wheel 后离线安装 |
| wheel 架构错误 | x86_64 / cp310 wheel 不能安装 | 架构和 Python 版本不匹配 | 使用 `cp38 + aarch64` |
| ROS2 通信不稳定 | DDS/RTPS 报错 | DDS 网卡或配置可能不正确 | 绑定 CycloneDDS 到 `enx00e04c36178d` |
| Docker 权限问题 | Docker API permission denied | 用户不在 docker 组 | `sudo usermod -aG docker $USER` 后重新登录 |
| Docker 拉取失败 | 无法下载镜像 | 代理或网络影响 | 关闭代理后验证成功 |
| Dev Containers 慢 | Installing VS Code Server 慢 | 容器网络或下载源问题 | 放弃容器主流程，改为本地写代码 |
| Codex 插件卡住 | 登录/网络不稳定 | 插件和网络环境不稳定 | 不依赖 Codex，改为手动维护代码 |
| NoMachine 下载失败 | 域名解析失败 | 远程端无外网/DNS 异常 | 本地下载后 `scp` 传输 |
| VS Code Server 下载慢 | 卡在 `Downloading VS Code Server...` | 远程端无法联网下载 | 设置 `remote.SSH.localServerDownload = always` |
| 可视化需求 | SSH 只有命令行 | 需要图形远程方案 | 简单 GUI 用 X11，完整桌面用 NoMachine |
