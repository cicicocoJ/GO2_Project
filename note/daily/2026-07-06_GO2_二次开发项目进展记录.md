### 3.2 backend_client 启动问题修复

之前 `backend_client_node` 没有正常出现，日志中出现：

```text
exec: BACKEND_PORT=8000: not found
```

原因是脚本中把环境变量赋值写成了命令参数，导致 `exec` 把 `BACKEND_PORT=8000` 当成命令执行。

修复方向：

```bash
env BACKEND_PORT=8000 bash scripts/start_bridge.sh 192.168.7.124
```

修复后，一键启动中 `backend_client_node` 能正常拉起。

### 3.3 重复节点问题排查与解决

启动过程中曾出现大量重复节点，例如：

```text
/backend_client_node
/backend_client_node
/backend_command_handler_node
/backend_command_handler_node
/backend_command_handler_node
/camera_capture_node
/camera_capture_node
```

原因：

```text
stop_robot_side.sh 主要依赖 pid 文件停止进程
历史遗留或手动启动的进程没有 pid 文件记录
导致 stop 脚本没有清干净
反复 start 后节点不断叠加
```

排查时通过：

```bash
ps -ef | grep -E "backend_client_node|backend_command_handler_node|camera_capture_node|go2_state_reader_node" | grep -v grep
```

确认有真实残留进程，而不是 ROS2 daemon 缓存。

最终处理方式：

```text
按 PID 强制 kill 残留进程
在 start_robot_side.sh / stop_robot_side.sh 中加入兜底清理逻辑
启动前先清理旧进程
停止后再按进程名清理残留
```

修复后节点恢复正常：

```text
/backend_client_node
/backend_command_handler_node
/camera_capture_node
/go2_state_reader_node
/hesai_ros_driver/hesai_ros_driver_node
```

## 4. Dashboard 与视频流修复

### 4.1 Dashboard 视频不显示问题

一开始视频流网页可以打开，但 Dashboard 中没有画面。排查发现 `server.py` 中视频地址写死为旧地址：

```text
http://192.168.123.18:8081/video_feed
```

但当前笔记本通过 WiFi 访问 Jetson，应使用：

```text
http://192.168.7.149:8081/video_feed
```

### 4.2 改为读取配置

为了避免以后换网络还要改 `server.py`，将视频地址改为从环境变量读取：

```text
JETSON_IP
JETSON_VIDEO_PORT
GO2_VIDEO_STREAM_URL
```

目标逻辑：

```text
go2_network.env
  ↓
start_backend.sh export JETSON_IP / JETSON_VIDEO_PORT / GO2_VIDEO_STREAM_URL
  ↓
server.py 生成 Dashboard 视频地址
  ↓
Dashboard 显示当前 Jetson 视频流
```

### 4.3 8081 视频服务排查

曾出现：

```text
curl: Failed to connect to 192.168.7.149 port 8081: Connection refused
```

排查发现：

```text
8081 没有服务监听
camera_stream 没启动或启动后崩溃
camera_capture_node 多个重复进程可能占用了 /dev/video4
```

检查命令：

```bash
ssh unitree@192.168.7.149 '
ss -ltnp | grep 8081 || true
ps -ef | grep -E "go2_camera_stream_server|camera_capture_node" | grep -v grep || true
tail -n 120 /home/unitree/go2_bridge_ws/logs/runtime/camera_stream.log || true
'
```

最终通过清理旧进程并重新一键启动，Dashboard 画面恢复。

### 4.4 视频卡顿现象

Dashboard 视频已经显示，但画面较卡。原因可能包括：

```text
D435i 视频流 MJPEG 压缩和传输占用带宽
手机热点 / WiFi 不稳定
Jetson 同时运行多个 ROS2 节点、雷达驱动、视频流服务
浏览器端 MJPEG 解码性能一般
```

后续可优化方向：

```text
降低视频分辨率
降低 MJPEG 帧率
降低 JPEG quality
必要时将视频流与雷达/SLAM调试分开运行
```

## 6. XT-16 雷达与 RViz2 排查

### 6.1 RViz2 初始启动失败

笔记本运行：

```bash
bash scripts/start_lidar_rviz_laptop.sh
```

最初报错：

```text
rviz2: enx00e04c36178d: does not match an available interface
rmw_create_node: failed to create domain
```

原因：

```text
start_lidar_rviz_laptop.sh 中写死了旧有线网卡 enx00e04c36178d
当前使用的是 WiFi 网卡 wlp2s0
```

修复方向：

```text
脚本读取 go2_network.env
LAPTOP_DDS_IFACE=wlp2s0
不再写死 enx00e04c36178d
```

修复后 RViz2 能正常打开：

```text
Interface: wlp2s0
OpenGl version: 4.6
```

### 6.2 XT-16 驱动状态确认

一开始怀疑 XT-16 驱动没起来，但通过日志确认驱动正常：

```text
raw frame:5536 points:64000 packet:500
raw frame:5537 points:64000 packet:500
```

进程也存在：

```text
ros2 launch hesai_ros_driver start.py
hesai_ros_driver_node
```

Jetson 本机能看到：

```text
/lidar_points
/lidar_imu
/lidar_packets_loss
```

结论：

```text
XT-16 驱动正常
雷达数据正常
问题主要在笔记本与 Jetson 之间的 DDS 发现/传输网络
```

### 6.3 WiFi 下完整点云传输问题

由于 XT-16 一帧约：

```text
64000 points
500 packets
约 10Hz
```

完整 `/lidar_points` 通过 WiFi / 手机热点传输数据量较大，容易出现：

```text
ros2 topic list 卡住
RViz2 打开慢
点云画面卡顿
DDS 多播发现不稳定
```

因此今天形成新的判断：

```text
日常 Dashboard / 视频 / 状态可以走 WiFi
建图 / RViz2 / 点云建议走有线
```

## 8. 关于是否让 Jetson 本地建图的讨论

讨论过一种方案：

```text
Jetson 本地运行 XT-16、pointcloud_to_laserscan、slam_toolbox、RViz2
笔记本用 NoMachine 远程查看
手柄操作机器狗移动建图
```

这个方案可行，优点是：

```text
不需要把完整点云传到笔记本
避开 WiFi DDS 发现问题
所有 SLAM 数据都在 Jetson 本地闭环
```

但缺点是：

```text
Jetson 跑 RViz2 有图形压力
点云可视化和 SLAM 同时运行可能卡顿
NoMachine 远程桌面也会占用一定资源
```

因此最终决定：

```text
如果 Jetson 跑 RViz2 压力较大，建图阶段仍推荐 Jetson 和笔记本之间插网线
笔记本跑 RViz2 和 SLAM
Jetson 跑雷达驱动和 GO2 通信
```
