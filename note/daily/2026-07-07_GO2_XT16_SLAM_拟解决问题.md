# 2026-07-07 GO2 + XT-16 SLAM 拟解决问题记录

## 一、当前阶段结论

截至目前，GO2 + Hesai XT-16 的 2D SLAM 初步流程已经打通，已经完成：

```text
XT-16 /lidar_points
    -> pointcloud_to_laserscan
    -> /scan
    -> scan_stamp_relay
    -> /scan_slam
    -> slam_toolbox
    -> /map
    -> map_saver_cli 保存 yaml + pgm
    -> map_server 重新加载地图
    -> RViz2 显示静态地图
```

说明目前已经具备：

1. 外接 XT-16 雷达数据接入能力；
2. 点云转 2D LaserScan 的能力；
3. slam_toolbox 在线建图能力；
4. 地图保存与重新加载能力；
5. 基于有线雷达链路的 SLAM 测试环境。

但是当前生成的地图质量仍然较差，主要表现为：

- 地图边缘不清晰；
- 墙体、障碍物存在重影；
- 转弯时地图容易错位；
- 大范围建图时局部结构容易漂移；
- 直线运动相对可用，但转向后图像质量明显下降；
- 当前流程更像“能跑通 Demo”，还没有达到高质量建图和导航地图要求。

因此，2026-07-07 的重点不是继续扩大地图范围，而是系统排查影响地图质量的关键问题。

## 二、核心问题 1：`base_link -> hesai_lidar` 外参仍是临时值

### 1. 问题描述

当前使用的 XT-16 外参是临时估计值：

```text
base_link -> hesai_lidar
x = 0.20
y = 0.00
z = 0.25
roll = 0.00
pitch = 0.00
yaw = 0.00
```

该外参只是为了让 TF 链路跑通，并不是实测标定结果。

如果雷达安装方向与 `base_link` 实际存在偏角，尤其是 `yaw` 偏差，即使只有几度，也会导致：

- 直线时地图勉强可用；
- 转弯后墙体出现双层；
- `/scan_slam` 与 `/map` 边缘对不上；
- 地图整体旋转、错位；
- 大范围地图逐渐累积误差。

### 2. 拟解决目标

确认 XT-16 相对于 GO2 机体的真实安装位置和朝向，重点校准：

```text
base_link -> hesai_lidar yaw
```

其次再校准：

```text
x / y / z / roll / pitch
```

### 3. 拟测试方法

选择一面直墙或走廊环境：

1. 机器人静止，观察 `/scan_slam` 是否与墙体方向一致；
2. 机器人缓慢原地转动 15°～30°；
3. 观察墙体是否变成双层；
4. 多次微调 `yaw`，观察地图重影是否减轻。

### 4. 涉及命令

查看当前 TF：

```bash
source ~/GO2_Project/go2_ros_env_lidar_wired.sh
ros2 run tf2_ros tf2_echo base_link hesai_lidar
```

修改启动脚本中的静态 TF：

```bash
gedit ~/GO2_Project/go2_bridge_ws/scripts/start_2d_slam_laptop.sh
```

重点修改：

```bash
ros2 run tf2_ros static_transform_publisher \
  --x 0.20 \
  --y 0.00 \
  --z 0.25 \
  --roll 0.00 \
  --pitch 0.00 \
  --yaw 0.00 \
  --frame-id base_link \
  --child-frame-id hesai_lidar
```

## 三、核心问题 2：当前时间戳处理只是临时方案

### 1. 问题描述

为了解决早期 `slam_toolbox` 报错：

```text
the timestamp on the message is earlier than all the data in the transform cache
```

当前采用了临时处理方法：

```text
/scan -> /scan_slam
header.stamp = now()
```

同时：

```text
/utlidar/robot_odom -> odom -> base_link
TF stamp = now()
```

该方法可以让系统跑通，但并不是严格时间同步。

实际系统中存在多个时间：

```text
雷达真实采样时间
点云转换为 scan 的时间
scan_stamp_relay 发布时间
odom 估计时间
TF 发布时间
DDS 传输延迟
```

静止或低速直线时影响较小；一旦转弯，时间误差会转化为角度误差，导致地图重影和错位。

### 2. 拟解决目标

梳理当前各话题时间戳来源，判断是否可以逐步减少 `now()` hack。

重点确认：

- `/lidar_points` 的时间戳是否稳定；
- `/scan` 的时间戳是否继承自点云；
- `/scan_slam` 当前发布时间是否合理；
- `/utlidar/robot_odom` 的时间戳是否可用；
- TF 时间是否与 scan 匹配。

### 3. 涉及命令

查看 `/lidar_points` 时间戳：

```bash
source ~/GO2_Project/go2_ros_env_lidar_wired.sh
ros2 topic echo /lidar_points --once | sed -n '1,30p'
```

查看 `/scan_slam` 时间戳：

```bash
ros2 topic echo /scan_slam --once | sed -n '1,30p'
```

查看 `/utlidar/robot_odom` 时间戳：

```bash
ros2 topic echo /utlidar/robot_odom --once | sed -n '1,40p'
```

查看 TF：

```bash
ros2 run tf2_ros tf2_echo odom base_link
```

## 四、核心问题 3：XT-16 旋转雷达点云未做完整运动去畸变

### 1. 问题描述

XT-16 是旋转式 3D 激光雷达，一帧点云不是在同一时刻完成采样，而是在一段时间内旋转扫描形成。

机器人直线低速运动时，点云畸变不明显；但转弯时，雷达一帧扫描期间机器人姿态发生变化，会导致：

- 点云内部存在运动畸变；
- 转成 `/scan` 后边界变形；
- slam_toolbox 的 scan matching 误差变大；
- 转弯处地图出现重影；
- 速度越快，畸变越明显。

### 2. 拟解决目标

先通过低速测试确认运动畸变影响程度，再决定是否需要引入 3D LIO 或点云去畸变方案。

### 3. 拟测试方法

对比三组运动：

#### 测试 A：只直线运动

```text
静止 10 秒
-> 前进 1～2 米
-> 停止
-> 保存地图
```

观察地图是否仍然重影。

#### 测试 B：极慢转弯

```text
每次转 10°～15°
每次转完停 2 秒
总共转 45°～90°
```

如果地图明显改善，说明问题主要来自转弯畸变和时间同步。

#### 测试 C：正常速度转弯

与测试 B 对比，如果正常速度明显变差，则进一步确认运动畸变影响。

### 4. 运动控制建议

建图测试时建议：

```text
前进速度：0.10～0.15 m/s
转向速度：0.10～0.20 rad/s
尽量少横移
每次动作后停 1～2 秒
```

## 五、核心问题 4：GO2 四足平台本身姿态晃动较大

### 1. 问题描述

GO2 是四足机器人，不是稳定的轮式底盘。行走过程中存在：

- 机身上下起伏；
- roll / pitch 摆动；
- 足端冲击；
- 转弯时姿态变化更明显；
- 步态导致雷达平面不稳定。

当前做法是从 XT-16 的 3D 点云中截取一个高度范围：

```text
z in [-0.50, 0.50]
```

再转换成 2D LaserScan。

但当机身 roll/pitch 变化时，这个切片并不是严格稳定的水平切片，同一面墙在不同姿态下可能被截到不同高度，导致 `/scan` 边界不稳定。

### 2. 拟解决目标

评估不同高度切片参数对 `/scan` 稳定性的影响，寻找更适合 GO2 行走姿态的 2D scan 参数。

### 3. 拟测试参数

可测试以下几组：

```text
方案 1：min_height=-0.30, max_height=0.30
方案 2：min_height=-0.50, max_height=0.50
方案 3：min_height=-0.20, max_height=0.40
方案 4：min_height=-0.10, max_height=0.50
```

同时观察：

- `/scan_slam` 点数是否足够；
- 墙体边缘是否稳定；
- 地图是否重影；
- 地面点是否影响建图；
- 桌腿、门框、柱子等特征是否保留。

## 六、核心问题 5：`/utlidar/robot_odom` 与外接 XT-16 可能不同源不同步

### 1. 问题描述

当前里程计来源是：

```text
/utlidar/robot_odom
```

该里程计来自 Unitree 官方内部雷达/SLAM/定位系统，未必与外接 XT-16 严格同步，也未必使用同一套传感器坐标定义。

它可以帮助系统跑通，但未必能支撑高质量外接雷达 SLAM。

潜在问题包括：

- odom 与 XT-16 scan 时间不一致；
- odom 有延迟；
- odom 自身有跳变或修正；
- odom frame 与外接雷达 frame 没有严格标定；
- 转弯时 yaw 变化与 scan 匹配结果不一致。

### 2. 拟解决目标

验证 `/utlidar/robot_odom` 的方向、连续性和稳定性。

### 3. 拟测试方法

#### 测试方向是否符合 ROS 约定

ROS 常用约定：

```text
x：前方为正
y：左方为正
yaw：左转为正
```

执行：

```bash
source ~/GO2_Project/go2_ros_env_lidar_wired.sh
timeout 20s ros2 run tf2_ros tf2_echo odom base_link
```

然后分别执行：

```text
前进
后退
左转
右转
```

观察 TF 变化方向是否正确。

#### 测试 odom 是否跳变

机器人静止 30 秒，记录：

```bash
ros2 topic echo /utlidar/robot_odom
```

观察位置和姿态是否明显漂移或跳变。

## 七、核心问题 6：slam_toolbox 参数仍需针对四足 + 3D 雷达调优

### 1. 当前降负载参数

为了解决：

```text
discarding message because the queue is full
```

当前已经做了降负载处理：

```text
/scan_slam 降频
angle_increment 增大
range_max 降低
map_update_interval 增大
暂时关闭 loop closing
```

这解决了队列刷屏问题，但地图质量仍然不理想。

### 2. 拟解决目标

在系统稳定运行后，再逐步优化 slam_toolbox 参数，而不是一次性大改。

重点关注：

```text
minimum_time_interval
map_update_interval
minimum_travel_distance
minimum_travel_heading
max_laser_range
scan_buffer_size
do_loop_closing
```

### 3. 当前建议

先以稳定为主：

```yaml
minimum_time_interval: 0.30
map_update_interval: 4.0
max_laser_range: 12.0
do_loop_closing: false
```

如果小范围地图稳定，再逐步恢复：

```yaml
do_loop_closing: true
map_update_interval: 2.0
```

## 八、核心问题 7：是否引入 Fast-LIO / Fast-LIO2

### 1. 问题背景

当前 2D slam_toolbox 链路可以跑通，但四足机器人 + 3D 雷达 + 外接雷达时间同步问题，会限制地图质量。

Fast-LIO / Fast-LIO2 可以作为下一阶段 3D LiDAR-Inertial Odometry 方案，用于提供更高质量的局部里程计。

但需要注意：Fast-LIO 本身主要是前端里程计与局部建图，不是完整全局 SLAM 后端。它通常不直接解决：

- 全局回环；
- 全局图优化；
- 稳定 `map -> odom`；
- Nav2 2D 地图格式；
- 长时间全局一致性。

### 2. Fast-LIO 在本项目中的潜在角色

未来结构可以是：

```text
XT-16 点云 + IMU
    -> Fast-LIO2
    -> odom -> base_link
    -> 局部 3D map
    -> 后端回环 / GICP / GTSAM，可选
    -> map -> odom
    -> 2D 投影地图 / Nav2 / 3D 规划
```

也就是说：

```text
Fast-LIO 负责局部连续位姿
后端负责全局一致性
Nav2 / planner 负责导航任务
```

### 3. 引入 Fast-LIO 前必须确认的问题

需要先确认：

```text
/lidar_points 是否有每点时间戳？
timestamp 字段单位是什么？
ring 字段是否正确？
可用 IMU 是哪个？
/lidar_imu 是否可用？
/utlidar/imu 是否可用？
/lf/lowstate 中 IMU 频率是多少？
雷达和 IMU 是否时间同步？
LiDAR -> IMU 外参如何标定？
```

如果这些问题没有解决，Fast-LIO 也可能在转弯时发生漂移或炸图。

## 九、7.7 推荐排查顺序

建议按照以下顺序推进，避免直接进入复杂 Fast-LIO 后端开发。

### Step 1：复现问题

用当前脚本重新建一张小地图，记录问题现象：

```text
直线是否可用？
转弯是否重影？
重影是固定方向还是随机？
地图是否整体旋转？
/scan_slam 是否贴合墙体？
```

### Step 2：只做直线测试

目标：判断基础 scan 和 odom 是否可靠。

### Step 3：极慢转弯测试

目标：判断是否主要由运动畸变 / 时间同步导致。

### Step 4：调整 `base_link -> hesai_lidar yaw`

目标：优先消除转弯重影。

### Step 5：测试不同高度切片

目标：找到更稳定的 `/scan` 生成参数。

### Step 6：验证 `/utlidar/robot_odom`

目标：判断当前 odom 是否适合继续作为 slam_toolbox 输入。

### Step 7：录制 rosbag

目标：为后续离线复盘和 Fast-LIO 适配做数据准备。

```bash
source ~/GO2_Project/go2_ros_env_lidar_wired.sh

mkdir -p ~/GO2_Project/rosbags
cd ~/GO2_Project/rosbags

  /lidar_points \
  /scan \
  /scan_slam \
```

## 十一、阶段性判断

当前项目已经完成了从“雷达数据接入”到“地图保存/加载”的流程闭环。

但地图质量问题说明：

```text
流程跑通 ≠ 输入数据质量满足高质量 SLAM
```

下一阶段的关键不是增加功能，而是把以下基础问题逐一确认：

```text
外参是否准确
时间戳是否合理
点云是否畸变
odom 是否可靠
四足平台运动是否过快
2D 切片是否稳定
```

只有这些问题得到控制后，再进入 AMCL、Nav2 或 Fast-LIO2 才更稳妥。
