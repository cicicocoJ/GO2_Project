## 二、当前项目总体架构理解

今天进一步明确了 GO2 二次开发中的网络与通信架构。

### 1. GO2 与 Jetson / NX 的关系

GO2 本体是机器狗自身的运动控制与传感系统；Jetson / NX 是额外挂载在机器狗上的边缘计算设备，用于运行二次开发程序、状态读取节点、后台中转节点、算法模块等。

可以理解为：

```text
GO2 本体：负责运动控制、底层状态、传感器数据
Jetson / NX：负责二次开发、中转通信、任务逻辑、算法部署
```

### 2. 有线 DDS 与无线控制的分工

结合开发者反馈，今天进一步确认：Unitree DDS 不适合直接通过无线传播，也不建议把无线网卡强行加入 192.168.123.x 网段。

正确架构是：

```text
GO2 本体
    ↓ 有线 DDS / 192.168.123.x
Jetson / NX
    ↓ WebSocket / HTTP / MQTT / WebRTC
外部后台 / 笔记本 / 手机 / 云端
```

其中：

```text
DDS：只在 GO2 与 Jetson / NX 内部有线网络中流转
无线：只传低频控制命令、状态摘要、任务信息、后台交互信息
```

因此，当前开发的 `backend_client_node` 正是系统中的中转层。

## 三、今日完成的主要工作

## 3.1 确认 Unitree 真实状态话题

在 Jetson 上通过 `ros2 topic list -t` 确认到了 GO2 的真实状态话题。

主要话题包括：

```text
/lf/sportmodestate [unitree_go/msg/SportModeState]
/sportmodestate    [unitree_go/msg/SportModeState]

/lf/lowstate       [unitree_go/msg/LowState]
/lowstate          [unitree_go/msg/LowState]
```

最终选择优先使用：

```text
/lf/sportmodestate
/lf/lowstate
```

原因是 `lf` 表示 low frequency，适合作为状态读取来源。

---

## 3.2 解决 Unitree 消息包缺失问题

最初执行：

```bash
ros2 topic echo /lf/sportmodestate
```

出现错误：

```text
ModuleNotFoundError: No module named 'unitree_go'
```

这说明 DDS 话题本身已经存在，但当前 ROS2 Python 环境中缺少 `unitree_go` 消息包，导致无法反序列化消息内容。

今天明确了一个关键认识：

```text
DDS 负责“传输数据”
unitree_go / unitree_api 消息包负责“理解数据结构”
```

也就是说，能看到 topic 不等于能解析 topic。要想读取 `SportModeState` 和 `LowState`，必须让 ROS2 环境中具备对应消息定义。

随后补齐 / 编译 Unitree 消息环境后，`sportmodestate` 和 `lowstate` 均可以正常输出。

## 3.3 新增 go2_state_reader_node

今天新增了真实 GO2 状态读取节点：

```text
go2_state_reader_node
```

该节点主要功能如下：

```text
订阅：
/lf/sportmodestate
/lf/lowstate

发布：
/inspection_state
/go2_status_json
```

其中：

```text
/inspection_state
```

用于兼容已有的 `backend_client_node`，输出简单状态，例如：

```text
INIT
IDLE
PATROLLING
ERROR
```

而：

```text
/go2_status_json
```

用于输出更完整的 GO2 状态 JSON，包括速度、电量、位置、姿态、状态新鲜度等信息。

### 节点状态估计逻辑

第一版状态估计规则为：

```text
未收到 sportmodestate        → INIT
sportmodestate 超时           → ERROR
速度模长大于阈值              → PATROLLING
速度模长小于阈值              → IDLE
```

这里的 `PATROLLING` 暂时表示机器人处于运动状态。后续接入巡检任务状态机后，可以进一步区分：

```text
MOVING
PATROLLING
PAUSED
STOPPED
EMERGENCY_STOP
```

## 3.5 升级 backend_client_node

在 `go2_state_reader_node` 成功运行后，今天继续升级了 `backend_client_node`。

原来的 `backend_client_node` 只订阅：

```text
/inspection_state
```

因此后台只能看到简单状态：

```json
{
  "state": "IDLE",
  "battery": null,
  "current_checkpoint": null
}
```

升级后，`backend_client_node` 同时订阅：

```text
/inspection_state
/go2_status_json
```

然后将两个状态合并，上传给 FastAPI 后台。

升级后的上传内容包括：

```text
state
battery
speed_norm
position
velocity
yaw_speed
go2_simple_state
sport_received
lowstate_received
go2_status_received
go2_status
```

其中 `go2_status` 保留完整 JSON，方便后续后台继续扩展显示更多机器人状态。
